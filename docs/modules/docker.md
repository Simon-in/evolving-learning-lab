# Docker 组件学习文档

## 1. 组件定位

Docker 是你学习大数据组件时的环境底座，核心作用是：

- 快速搭建本地可复现的多组件环境（Hadoop/Hive/Spark 等）。
- 降低环境安装门槛，避免被系统依赖和版本冲突卡住。
- 支持后续把学习项目迁移到云上容器化运行思路。

在你的学习路线中，Docker 的价值是：先解决“怎么稳定跑起来”，再专注组件本身能力。

## 2. 核心概念

### 2.1 镜像与容器

- Image：运行环境模板（包含依赖、配置、启动命令）。
- Container：镜像的运行实例。
- Registry：镜像仓库（如 Docker Hub）。

你至少要能回答：

1. 镜像和容器的区别是什么？
2. 为什么同一个镜像可以启动多个容器？

### 2.2 数据与网络

- Volume：持久化数据，容器删除后数据仍可保留。
- Bind Mount：把宿主机目录映射进容器，便于开发调试。
- Bridge Network：让多个容器通过服务名互联。

你至少要能回答：

1. 为什么数据库/元数据服务必须做持久化？
2. 多容器之间如何互相访问？

### 2.3 编排（Docker Compose）

- `docker compose up -d`：启动一组关联服务。
- `docker compose ps`：查看服务状态。
- `docker compose logs -f <service>`：查看服务日志。
- `docker compose down`：停止并清理容器网络。

你至少要能回答：

1. 为什么大数据学习项目更适合用 Compose 管理？
2. 服务启动顺序与依赖关系如何处理？

## 3. 本阶段实操任务（最小闭环）

### 任务 A：单容器基础操作

- 拉取镜像、启动容器、进入容器、停止与删除容器。
- 目标：熟悉容器生命周期命令。

### 任务 B：多容器编排

- 用 Compose 启动至少 2 个有依赖关系的服务。
- 验证容器间网络互通和服务名访问。
- 目标：掌握多服务协作的基础方法。

### 任务 C：日志与故障排查

- 人为制造一个错误配置（端口冲突或环境变量错误）。
- 通过 `docker compose logs` 定位并修复问题。
- 目标：建立“看日志 -> 定位问题 -> 修复”的操作闭环。

## 4. 常见问题与排查

### 4.1 容器启动失败

常见原因：
- 端口被占用
- 环境变量缺失
- 镜像版本不兼容

排查思路：
- 先看 `docker compose ps` 状态，再看对应服务日志。

### 4.2 数据丢失

常见原因：
- 未配置 Volume
- `down -v` 误删卷

排查思路：
- 区分临时数据和持久化数据，关键服务默认绑定 Volume。

## 5. 云厂商能力映射

结合你的 AWS 背景，可做如下映射：

- Docker 容器化能力 -> 云上容器服务运行能力（ECS/EKS/同类平台）
- Compose 多服务编排思维 -> 云上编排与服务治理思维
- 本地镜像与配置管理 -> 云上 CI/CD 与镜像仓库管理能力

重点是迁移“环境标准化与部署自动化”能力。

## 6. 面试可讲述模板

1. 我使用 Docker 标准化了大数据学习环境，避免本地依赖冲突。
2. 我用 Compose 管理多服务启动与依赖，保障了链路可复现。
3. 我能通过日志快速定位容器启动失败、网络和配置问题。
4. 这些能力可直接迁移到云上容器化部署场景。

## 7. 本组件完成标准（DoD）

满足以下条件即可认为 Docker 阶段完成：

- 能独立完成镜像、容器、网络、卷的基础操作。
- 能用 Compose 启动并维护一套多服务环境。
- 能独立排查常见启动故障和数据持久化问题。
- 能将本地容器化经验映射到云上部署能力。
