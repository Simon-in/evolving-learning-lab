# Hive 组件学习文档

## 1. 组件定位

Hive 是离线数仓中最常见的 SQL 数据仓库组件，核心作用是：

- 把 HDFS 上的数据组织成可查询的表结构。
- 提供统一 SQL 入口，支持数仓分层建模与批处理开发。
- 作为 Spark SQL、Trino 等引擎的重要元数据基础。

在你的学习路线中，Hive 的价值是：把“会写 SQL”升级为“会做数仓建模与离线产出”。

## 2. 核心概念

### 2.1 元数据与表

- Metastore：存储库、表、分区、字段等元数据。
- External Table：只管理元数据，不托管底层数据生命周期。
- Managed Table：由 Hive 管理数据与元数据生命周期。

你至少要能回答：

1. 外部表和内部表在生产中的使用边界是什么？
2. 为什么“删表是否删数据”会影响数据安全策略？

### 2.2 分区与分桶

- Partition：按业务字段切分目录，减少扫描范围。
- Bucket：按哈希将数据分桶，优化部分 Join/采样场景。
- Partition Pruning：查询时只扫命中的分区。

你至少要能回答：

1. 分区字段该如何选？
2. 什么时候只分区就够，什么时候考虑分桶？
3. 分区过多会带来什么问题？

### 2.3 执行与优化

- 执行计划：用 `EXPLAIN` 看 SQL 物理执行路径。
- 常见代价来源：全表扫描、大量 Shuffle、数据倾斜。
- 优化手段：分区裁剪、字段裁剪、Join 顺序优化。

你至少要能回答：

1. 一条慢 SQL 如何系统定位瓶颈？
2. 为什么“少扫数据”几乎总是第一优化原则？

## 4. 常见问题与排查

### 4.1 查询慢

常见原因：
- 没命中分区
- 扫描字段过多
- Join 键选择不合理

排查思路：
- 先看 `EXPLAIN`，再看是否发生全表扫描和大规模 Shuffle。

### 4.2 分区管理混乱

常见原因：
- 分区字段与业务查询习惯不一致
- 分区数量失控
- 数据延迟写入导致分区缺失

排查思路：
- 先统一分区策略，再补齐分区巡检和数据完整性检查。
