# Spark 组件学习文档

## 1. 组件定位

Spark 是离线数仓中的核心计算引擎，主要用于：

- 大规模数据的批处理计算与转换。
- 基于 DataFrame/SQL 的指标加工与宽表构建。
- 与 Hive 元数据体系配合，形成标准化离线生产链路。

在你的学习路线中，Spark 的价值是：把“会写 Hive SQL”升级为“会做高性能离线数据计算”。

## 2. 核心概念

### 2.1 DataFrame 与执行模型

- DataFrame：结构化数据抽象，适合工程化开发。
- Lazy Evaluation：转换操作不会立即执行，直到触发 Action。
- DAG：Spark 会将任务拆解成阶段（Stage）与任务（Task）执行。

你至少要能回答：

1. 为什么 Spark 可以延迟执行？这对优化有什么好处？
2. 一个 DataFrame 作业是如何被拆成 Stage 的？

### 2.2 Shuffle 与 Join

- Shuffle：跨分区重分布数据，通常是性能瓶颈来源。
- 常见 Join 策略：Broadcast Join、Shuffle Hash Join、Sort Merge Join。
- 数据倾斜：少量 Key 数据量过大，导致任务“拖尾”。

你至少要能回答：

1. 哪些操作会触发 Shuffle？
2. 什么时候优先用 Broadcast Join？
3. 遇到数据倾斜时如何处理？

### 2.3 资源与参数

- Executor：执行计算任务的进程。
- `executor-memory` / `executor-cores`：影响并行度与内存稳定性。
- `shuffle.partitions`：影响 Shuffle 后任务粒度与调度开销。

你至少要能回答：

1. 为什么“盲目加资源”不一定加速任务？
2. 参数调优的基本顺序应该是什么？

## 3. 本阶段实操任务（最小闭环）

### 任务 A：DataFrame 基础加工

- 读取明细数据，完成过滤、聚合、去重、窗口计算。
- 输出一张日粒度指标表。

目标：掌握 DataFrame API 与 SQL 结合方式。

### 任务 B：Join 策略实验

- 用两种以上 Join 策略处理同一业务问题。
- 对比执行时间与资源消耗。

目标：建立“业务场景 -> Join 策略”的选择能力。

### 任务 C：参数调优实践

- 调整 `executor-memory`、`executor-cores`、`shuffle.partitions`。
- 记录调优前后性能变化。

目标：形成可复用的 Spark 调优思路。

## 4. 常见问题与排查

### 4.1 作业慢或长尾严重

常见原因：
- Shuffle 过多
- 分区不均导致数据倾斜
- Join 策略不合适

排查思路：
- 先看 Spark UI 的 Stage/Task 分布，再定位倾斜与瓶颈算子。

### 4.2 OOM（内存溢出）

常见原因：
- 单分区数据过大
- 缓存策略不合理
- Executor 配置过小

排查思路：
- 优先优化数据分区与算子逻辑，再调资源参数，最后评估缓存必要性。

结合你的 AWS 背景，可做如下映射：

- 开源 Spark 作业开发能力 -> 云上托管 Spark 作业开发能力

- Spark + Hive 离线链路能力 -> 云上数仓离线生产能力

重点是迁移“计算与调优方法”，而不是迁移某个平台专有配置。
